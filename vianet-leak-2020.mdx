## The Incident
April 2020. Barely a month after Foodmandu, the other shoe dropped. **Vianet Communications**, one of the top ISPs in Nepal, leaked the personal data of over **170,000 customers**.

It wasn't a dark web sale at first; a Twitter user simply posted a sample: "Here are your emails, phone numbers, and biological details." Panic ensued.

## The Technical Flaw
It wasn't a complex zero-day. It was **BOLA (Broken Object Level Authorization)**—the modern plague of APIs.

### How It Works
The Vianet self-service portal had an API endpoint that looked something like this:
`GET /api/customer/1001/usage`

The problem? The server didn't check *who* was asking. It just checked if ID `1001` existed.
*   Attacker: "Give me data for user 1002."
*   Server: "Sure, here is Rabins' address."
*   Attacker: "User 1003?"
*   Server: "Here is Sunil's phone number."

This is currently the **#1 Vulnerability** on the **OWASP API Security Top 10**.

## The Aftermath
Vianet owned it, patched it, and apologized. But the damage was done. The incident, back-to-back with Foodmandu, forced the government to finally start drafting serious cyber laws.

## The Lessons
1.  **Stop Using Integers**: If your user IDs are `1, 2, 3`, you are making the hacker's job easy. Use **UUIDs** (like `a1b2-c3d4-e5f6`). Good luck guessing that.
2.  **API Gateways**: Don't rely on code-level checks alone. Use an API Gateway to enforce "User X can only see Resource X" rules globally.
3.  **Data Diet**: Why does the "Check Internet Balance" API need to return the user's home address and citizenship number? It doesn't. Stop sending extra data.
