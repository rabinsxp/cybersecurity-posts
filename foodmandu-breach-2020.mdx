## The Incident
In March 2020, Foodmandu faced a nightmare scenario. A hacker going by "Mr. Mugger" announced they had dumped the personal details of nearly **50,000 customers**.

The leak was extensive:
*   Full Names
*   Email Addresses
*   Phone Numbers
*   Exact Delivery Addresses

## What Went Wrong?
We never got a full, transparent post-mortem. However, looking at the patterns, this screams of a classic **Insecure Direct Object Reference (IDOR)** or a basic API enumeration flaw.

### The Attack Vector
It’s likely the attacker found an API endpoint like `/api/user/12345`. Instead of checking "Does the logged-in user own ID 12345?", the server simply returned the data. The attacker just had to write a script to count from 1 to 50,000 and scrape everything.

This is a textbook case of **Broken Access Control** (OWASP Top 10 A01).

## The Impact
Trust evaporated overnight. For users, having your phone number and home address leaked brings real physical risks, not just digital ones. For the Nepali tech scene, it was a harsh wake-up call: "Security through obscurity" is dead.

## The Fixes
1.  **Rate Limiting**: If someone asks for 10,000 user profiles in one minute, block them. It's that simple.
2.  **Authorization Checks**: Never trust the client ID. Always check `if (request.userId == session.userId)`.
3.  **VAPT**: A basic penetration test would have caught this in hour one. Don't skip it.
